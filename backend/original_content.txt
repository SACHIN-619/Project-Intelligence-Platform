22: 1. Executive Summary
23: Project Impact Intelligence (PII) is an AI-powered enterprise platform built to solve one of the most expensive problems in EPC (Engineering, Procurement, Construction) data centre delivery: information fragmentation. In a typical data centre construction project, specifications, vendor submittals, test records, RFIs, and change orders exist in disconnected silos — spreadsheets, PDFs, emails, and verbal handoffs — with no intelligence layer connecting them. This fragmentation results in critical path delay risks being discovered weeks after they have already locked in schedule slips.
24: PII unifies these fragmented data sources into a living intelligence layer that continuously understands the project state. Rather than being a simple document repository with a chatbot bolted on top, PII is architected around five integrated intelligence engines: Predictive Schedule Risk, Supply Chain Visibility, Commissioning Quality Assurance, Project Knowledge & RAG, and an underlying Knowledge Graph and AI Memory system that allows every module to reason about relationships between equipment, vendors, schedules, and risks — not just isolated facts.
25: The platform is built on FastAPI, PostgreSQL with the pgvector extension, and a resilient 3-layer AI fallback chain (Google Gemini → Groq LLaMA-3.3-70B → graceful degradation) that guarantees the system never crashes during a live demonstration, even under AI provider quota limits. The frontend is a Next.js 14 application offering a dark, mission-control-style interface designed around one guiding question per screen: what does this user need to decide right now?
26: 2. Problem Statement & Business Context
27: The ET AI Hackathon 2.0 Problem Statement 4 challenges teams to build an AI platform that unifies project documents, specifications, schedules, procurement data, and quality records into a living intelligence layer for EPC data centre delivery. The statement identifies five specific capability areas the platform must address. A single week of schedule delay on a ₹500 crore data centre project can cost several crores in penalty clauses, extended financing costs, and lost revenue from delayed go-live. PII is designed to reduce the manual coordination effort — measured in hours — which is the exact metric tracked and reported on the Admin dashboard.
28: 3. Solution Overview
29: PII reframes the challenge from 'build a chatbot over documents' to 'build the missing intelligence layer between fragmented EPC data sources.' The platform ingests project files, extracts structured knowledge, builds a relationship graph between entities (tasks, vendors, risks, equipment), and layers five specialised intelligence engines on top of that shared foundation.
30: The Core Ingestion Pipeline follows this workflow: Ingest → Parse → Standardise Schema → Build Entity Graph → Analyze CPM → Correlate Risks → Simulate Recovery → Learn Decision. Every uploaded file passes through this same pipeline. The result is that new capabilities (weather risk, vendor scoring, AI memory) can be added as consumers of the shared Knowledge Graph and Memory Service rather than as isolated, disconnected features.

65: 7. Core Innovation — The Living Intelligence Layer
66: Typical project tools treat documents as simple strings or PDF indexes. PII introduces a structured Relational Entity Graph where documents, tasks, vendors, and risks exist as interconnected nodes. This allows reasoning engines to trace the dependencies of a physical asset throughout the project timeline.
67: ✦ Cross-Project Vendor Scores: Aggregates schedule slips automatically and feeds them back into current task risks.
68: ✦ AI Decision Memory: Persists PM approvals/rejections to improve future recommendation algorithms.
69: ✦ Keyless Weather Agents: Open-Meteo API checks 14-day forecasts against outdoor-sensitive task dates.
70: 8. Feature Deep Dive
71: 8.1 Project Intelligence Ingestion Layer
72: Ingests CSV, XLSX, and PDF files. The Schema Mapper resolves naming variance (e.g. 'Sched_Start' vs. 'planned_start') using a 3-stage cascade (exact match, fuzzy Jaro-Winkler check, and semantic embedding distance) and caches the mappings per organisation for future imports.
73: 8.2 Predictive Schedule Risk Engine
74: Runs the Critical Path Method (CPM) to map task floats and critical milestones. Integrates task duration modeling using PERT-Beta probability distributions over 1,000 Monte Carlo runs to forecast milestone completion dates.
75: 8.3 Supply Chain Visibility & Vendor Reliability
76: Tracks delivery milestones for high-lead-time equipment. The Vendor Scorer computes on-time completion percentages across all projects. Low scores dynamically penalise schedule forecasts for tasks assigned to that supplier.
77: 8.4 Commissioning Quality Assurance Copilot
78: Ensures switchgear, UPS, HVAC, and generator startup tests are validated against Uptime Institute Tier IV standards, blocking downstream phases if prerequisite testing benchmarks are not met.
79: 8.5 Project Knowledge & RAG Assistant
80: Enables text-search and Q&A over project documentation. Cosine similarity over BGE-small embeddings finds specific passages, delivering answers complete with document name, page citations, and suggested follow-up prompts.
81: 9. 3-Layer AI Fallback Architecture
82: To prevent rate limit errors (like HTTP 429) from crashing live demonstrations or production workspaces, all AI API calls route through a silent middleware fallback chain:
83: 1. Layer 1: Google Gemini API (primary reasoning engine).
84: 2. Layer 2: Groq LLaMA-3.3-70B-Versatile (triggers if Gemini fails or rate-limits).
85: 3. Layer 3: Graceful Degradation (uses pre-formatted local response templates to ensure pipeline completion).
86: 10. Database Design
87: The relational database schema is designed for multi-tenant isolation. Submissions remain isolated by org_id.
88: 11. Security, RBAC & Audit
89: PII implements role-based access control (RBAC) both on the FastAPI routers and in the Next.js navigation menus. User sessions are verified via JWT tokens. Immutable logs of uploads, run simulations, and approved actions are saved to the audit_events table to provide accountability.

102: 15. Deployment Architecture
103: PII backend is deployed in a Docker environment containing FastAPI, a Redis task broker, and a PostgreSQL database. File storage is currently local but uses a StorageService abstraction supporting AWS S3 configurations.
